日志►log4j

log4j的学习

强哥说过，看日志很重要，以前学过一点log4j，忘得差不多了，今天再来学一下。以前一直认为，大脑的记忆力很重要，昨天看别人的一句话，觉得很有道理，记录一下，“大脑是用来思考的，不是用来记忆的”，我们站在巨人的肩膀上，不用去知道巨人内部是怎么工作的， 我们要爬上肩膀想办法看就好了。

言归正传。日志是干嘛的，就是记录应用程序的运行情况的信息，比如程序执行的过程、调试信息、错误信息、我们想要输出来的信息。

Log4j由三部分组成

一、Logger ，日志信息的优先级，表示了这条日志信息的重要程度，当设置一个记录器的优先级后，这个记录器只会输出比这个优先级高的日志，低的就不会输出来了。
fatal > error > warn > info > debug
fatal 表示致命错误，一旦出现就停止执行
error 表示一般错误，出现了程序还可继续执行
warn 警告，具有潜在的危害
info 指定能够突出在粗粒度级别的应用程序上运行情况的信息
debug 细粒度的最有用的应用程序调试信息

除了这五个常用的还有all、trace、off，all表示所有级别，trace表示比debug更加细粒度的信息，off表示关闭日志记录
off > fatal > error > warn > info > debug > trace > all

二、Appender ，日志信息的输出目的地，文件或者控制台
日志信息可以输出到控制台、某个文件、每天产生一个的文件、或者将日志信息以流格式发送到某个指定的地方，而我们选择的目的地，决定我们使用哪个类
1、org.apache.log4j.RollingFileAppender
2、org.apache.log4j.ConsoleAppender (控制台)
3、org.apache.log4j.FileAppender (文件)
4、org.apache.log4j.DailyRollingFileAppender (每天产生一个日志文件)
5、org.apache.log4j.WriterAppender (将日志信息以流格式发送到指定地方)

三、Layout ，日志信息的输出格式，日志信息的显示格式
%c{层数} logger的名称
%d 日志事件的时间
%m 日志信息
%n 换行
%level 日志级别
%p 日志的名字 （debug、fatal这样子的）
%t 当前生成日志事件的线程
还有一些其他的

如何使用log4j

首先我们定义一下配置文件，在配置文件里规定上面的三个东西，Logger日志种类、Appender输出目的地、Layout输出格式。我们需要配置一个根Logger，根logger指明打印的日志级别，然后指明使用的appender的名字。在定义的appender中，要指明appender的名字和leyout的名字，根据使用的类不同，其他的配置也不一样，看下：
1、首先配置Root的Logger
log4j.rootLogger = [ level ] , appenderName1, appenderName2, …
log4j.additivity.org.apache=false; 表示Logger不会在父Logger的appender里输出，默认为true。
level ：设定日志记录的最低级别，可设的值有OFF、FATAL、ERROR、WARN、INFO、DEBUG、ALL或者自定义的级别，Log4j建议只使用中间四个级别。通过在这里设定级别，可以控制应用程序中相应级别的日志信息的开关，比如在这里设定了INFO级别，则应用程序中所有DEBUG级别的日志信息将不会被打印出来。
appenderName：就是指定日志信息要输出到哪里。可以同时指定多个输出目的地，用逗号隔开。
例如：log4j.rootLogger＝INFO,A1,B2,C3

2、再配置Appender
log4j.appender.appenderName = className;
appenderName：自定义appderName，在log4j.rootLogger设置中使用；
className：可设值如下：
(1)org.apache.log4j.ConsoleAppender（控制台）
(2)org.apache.log4j.FileAppender（文件）
(3)org.apache.log4j.DailyRollingFileAppender（每天产生一个日志文件）
(4)org.apache.log4j.RollingFileAppender（文件大小到达指定尺寸的时候产生一个新的文件）
(5)org.apache.log4j.WriterAppender（将日志信息以流格式发送到任意指定的地方）
当选择的输出目的地不同的时候，配置也不相同
(1)ConsoleAppender选项：控制台
Threshold=WARN; 指定日志信息的最低输出级别，默认为DEBUG。
ImmediateFlush=true; 表示所有消息都会被立即输出，设为false则不输出，默认值是true。
Target=System.err; 默认值是System.out。
(2)FileAppender选项：文件
Threshold=WARN;
ImmediateFlush=true;
Append=false; true表示消息增加到指定文件中，false则将消息覆盖指定的文件内容，默认值是true。
File=D:/logs/logging.log4j; 指定消息输出到logging.log4j文件中。
(3)DailyRollingFileAppender选项：每天产生一个文件
Threshold=WARN;
ImmediateFlush=true;
Append=false;
File=D:/logs/logging.log4j;
DatePattern=’.’yyyy-MM; 每月滚动一次日志文件，即每月产生一个新的日志文件。当前月的日志文件名为logging.log4j，前一个月的日志文件名为logging.log4j.yyyy-MM。
另外，也可以指定按周、天、时、分等来滚动日志文件，对应的格式如下：
1)’.’yyyy-MM：每月
2)’.’yyyy-ww：每周
3)’.’yyyy-MM-dd：每天
4)’.’yyyy-MM-dd-a：每天两次
5)’.’yyyy-MM-dd-HH：每小时
6)’.’yyyy-MM-dd-HH-mm：每分钟
(4)RollingFileAppender选项：文件大小达到某个值的时候产生一个新的文件
Threshold=WARN;
ImmediateFlush=true;
Append=false;
File=D:/logs/logging.log4j;
MaxFileSize=100KB; 后缀可以是KB,MB或者GB。在日志文件到达该大小时，将会自动滚动，即将原来的内容移到logging.log4j.1文件中。
MaxBackupIndex=2; 指定可以产生的滚动文件的最大数，例如，设为2则可以产生logging.log4j.1，logging.log4j.2两个滚动文件和一个logging.log4j文件。

3、配置Layout
log4j.appender.appenderName.layout=className
className：可设值如下：
(1)org.apache.log4j.HTMLLayout（以HTML表格形式布局）
(2)org.apache.log4j.PatternLayout（可以灵活地指定布局模式）
(3)org.apache.log4j.SimpleLayout（包含日志信息的级别和信息字符串）
(4)org.apache.log4j.TTCCLayout（包含日志产生的时间、线程、类别等等信息）
(1)HTMLLayout选项：
LocationInfo=true：输出java文件名称和行号，默认值是false。
Title=My Logging： 默认值是Log4J Log Messages。
(2)PatternLayout选项：
ConversionPattern=%m%n：设定以怎样的格式显示消息。

参考自 http://www.cnblogs.com/ITtangtang/p/3926665.html 这里面有更详细的各种appender的配置

SpringBoot

Springboot的学习

首先简单学习了Spring的配置Bean的发展，XML配置到注解配置，再到现在的Java配置。
1、JDK1.5带来了注解支持，使得Spring2.x开始使用注解对Bean进行申明和注入。注解配置，大大减少了XML文件，也简化了项目的开发。只需要在使用Bean的地方写上注解就可以了。这时，我们遵循，应用的基本配置（数据源，资源文件）用xml，业务开发（Service注入Bean）用注解
2、Spring3.x开始提供了Java配置方式，这能更好的理解咱们配置的Bean。Spring4.x 和 SpringBoot都推荐使用java配置方式

那来学习一下Java配置长什么样子

1、注解：@Configuration 和 @Bean
作用： @Configuration作用于类上，表示这个类是一个java配置文件
@Bean 作用于方法上，表示这个方法返回一个Bean
2、简单来个粒子

其实现在还是不太明白，配置到底是在配置什么，Spring的原理还是不太清楚。毕设在即，先速成一下吧。

读取外部的资源配置文件

使用的注解： @PropertySource 和 @Value
@PropertySource 用于指定读取的配置文件
@Value 用于获取值，就是value写上配置文件中的键，然后把值给某某某
比如说，我们把数据源的配置写在了application.properties里面，那么怎么写配置呢

什么是动态语言，什么是静态语言？
静态语言是先编译后运行，动态语言是无需编译，直接运行。静态语言，比如Java、C，动态语言，比如Ruby、Groovy、Scala、Node.js、JS等等

随着动态语言的流行，Java开发显得笨重，配置、效率、部署流程等等都存在很大的问题，SpringBoot的生成，就是为了克服这些，SpringBoot让我们很容易创建一个独立运行的准生产级别的基于Spring框架的项目，SpringBoot会提供默认的配置，会解决部分依赖问题，它还内嵌了容器，框架强大吧。

创建SpringBoot应用

pom里面
首先 parent必须是 spring-boot-starter-parent,这里面的parent包含了大量的默认配置，大大简化了我们的开发（原来是在这里面）。
然后 添加web支持，是 spring-boot-starter-web
然后 添加springboot的插件 spring-boot-maven-plugin

SpringBoot项目会有一个入口类 ****Application.java，这个文件里面，有一个main方法，是标准的java应用程序的入口方法。

SpringApplication用来从java main方法启动一个spring应用，默认的启动步骤如下：
1)创建一个合适的ApplicationContext实例,这个实例取决于classpath。
2)注册一个CommandLinePropertySource，以spring属性的形式来暴露命令行参数。
3)刷新ApplicationContext，加载所有的单例bean。
4)触发所有的命令行CommanLineRunner来执行bean。
大部分场景下，可以从你的application的main方法中直接调用它的run()静态方法。就是最后一句没有被注释掉的代码。其实这个人解释的这四部分，我不太理解，先放着。

这是个入口类，还可以作为配置类，还能是一个controller
先讲一下@SpringBootApplication这个注解
@SpringBootApplication是SpringBoot的核心注解，是一个组合注解，咱么可以按ctrl打开看一下

@SpringBootApplication = @Target + @Retention + @Document + @Inherited + @SpringBootConfiguration + @EnableAutoConfiguratio + @ComponentScan
这里面 比较不基础的，有@SpringBootConfiguration、@EnableAutoConfiguration、@ComponentScan
1、@SpringBootConfiguration = @Target + @Retention + @Documented + @Configuration

2、@EnableAutoConfiguration 表示开启自动配置，SpringBoot会根据你在pom中引入什么依赖，他就会智能的帮你配置这些引入的jar的配置

3、@ComponentScan 默认扫描@SpringBootApplication所在类的同级目录以及它的子目录

综合来说，@SpringBootApplication这个注解的作用就是开启自动配置的注解，如果，你不想让你的某个模块采用自动配置，就这样
@SpringBootApplication(exclude = {XXXX.class})

好了，如果，你想修改默认配置，那么在全局配置文件中，修改，application.properties就是这个全局配置文件，往往，我们在这里采用“选择某个配置文件”的方式来写，就是说，在application.properties中指明你现在用的全局配置文件是什么，这么写spring.profiles.active=dev,然后，你就采用的是application-dev.properties这个配置文件，还可以是spring.profiles.active=pro，表示你用的是application-pro.properties，配置文件的类型，也可以是其他的，比如说yml类型的。全局配置文件中的键的意义有官方的标准。

算法►排序算法

各种排序

周四面试的时候，有排序的算法，唉，以前都没有注意，朋友来找我玩了几天，今天有空来总结一下。

实现一组无序数的升序排列，假设是10个整型数，a[10] = {34,56,23,1,54,76,34,53,27,9}

冒泡排序 bubble

时间复杂度：O(n²)
实现描述：每次将最大的数挪到未排序的所有数的最右边，从而实现排序

注意：双层for循环的范围要搞清楚

选择排序 select

算法描述：这个数组分有序区和无序区，初始状态是整个数组都是无序区，然后每次无序区选择一个最小的数将他放在有序区的最后边，末状态是整个数组都是有序区

把握好无序区和有序区即可

快速排序 Quick

java中Arrays类中的有关sort的排序用的就是快速排序，是目前已知的最快的排序方法，是冒泡排序的一种升级版
描述：通过一趟排序，将整个数列划分成两组，一组中的所有数要比另一组的数小，然后对每一组，再这样递归，最终形成一个有序数列

注意：一定要把算法的过程搞清楚，然后就是基数的选择，可以任意选择，也可以就选最左边的数。递归程序要想清楚，以后多过来看看想想。如何做到遍历一次就让基数的左边都比基数小，基数右边的数都比基数大，这个应该就是算法的精髓了。

插入排序 Insert

描述：也分有序和无序部分，就是将每个元素，插入有序部分中，最终形成整体有序

注意，在寻找插入位置时，有两种情况，一种是有序区的最左边的数比要插入的数大，一种是在有序区的中间位置找到比要插入的数小，这两种情况细枝末节不太一样，要分开考虑。

归并排序 Merge

描述：把一组分成两组，每一小组再分为两组，就这样一直递归分下去，最终呢，如果只有两个数，就按序排好返回去，只有一个不用排就返回去。这是末端情况，中间情况就是，得到了两个部分的排序后，将这两个小组排好序返回。这里排序呢，要注意一下，就是两个有序数列的合并。

Linux的文件系统

Linux下的文件系统

什么是文件系统？来理解一下

文件系统是基于被划分的存储设备上的一种文件的命名、存储、组织及读取的方法。

Linux文件系统的工作原理

文件系统通过为每个文件分配文件块的方式把数据存储在硬盘中。所以要维护每个文件的所有文件块的分配信息，这些信息也都存储在硬盘上。不同的文件系统，它的分配方式和读取文件块的方式是不一样的。
文件的块结构信息也称为meta-data（元信息），包括文件的权限、所有者、创建和访问和更改的时间等等这些与文件有关的信息。

数据结构►算法概念

算法和算法的评价

什么是算法

算法是对问题的求解的一种描述。通俗说，就是解决一个问题的方案。

算法的5个重要特性：有穷性、确定性、可行性、输入、输出
1、有穷性：一个算法在执行有限的步数之后一定要停止
2、确定性：算法的每一条指令必须要有明确的含义。也就是说，相同的输入必须得到相同的输出
3、可行性：算法中的每个操作都必须是基本的运算执行有限次就可完成的
4、输入：一个算法有m(m >= 0)个输入
5、输出：一个算法有n(n >= 1)个输出
