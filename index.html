字符编码规则的学习

什么是字符编码？

计算机的世界是二进制的，8bit也就是我们常说的一个字节，一般用两位十六进制数表示。那么计算机如何存储从键盘或者从文件等其他地方读取的字符呢，字符编码就是解决这个问题。我们将字符编码，字符一一映射为一个数值，然后计算机将值存储，当要显示在屏幕时按规则映射为对应的字符。

字符有哪些

字符包括电脑键盘上的字符、中文、其他国家的文字

编码规则

1、ASCII码

全称： American Standard Code for Information Interchange，美国信息交换标准代码
规则：用8bit表示字符，其中，最高位为0，其他7位对应的值映射到每个特定的字符
组成：ASCII码共包含128个字符，其中有95个可显示字符，33个不可显示字符
ASCII码对照表： http://ascii.911cha.com/

2、EASCII码

由ASCII码扩充而来，最高位也加入表示行列，扩充出来的编码用来表示表格符号、计算符号、希腊字母等等，但是还是无法显示中文等一些语言。目前这种编码很少使用了。

3、ISO/IEC 8859 编码

也是由ASCII码扩充而来，这种编码，是将字符集分组，一共有15组，这样表示：ISO/IEC 8859-n 这里面n的取值是从1到16除掉12，所以一共是15组，例如ISO/IEC 8859-1 ISO/IEC 8859-2,其实可以理解为有15中编码规则了，所以一共可以表示128*15这么多个字符，但是在实际写一个东西的时候，只能运用一组规则，也就是说，不能同时支持多个组里面的字符编码，还有个缺点是，这种字符集没有考虑亚洲文字，这也是个缺点吧

4、Unicode码(统一码)

还有种编码叫 ISO 10646编码标准，Unicode和ISO 10646是两个独立的尝试创立单一字符集的组织各自开发的，一个是International Standard Organization，一个是多语言软件制造商组成的统一码联盟。两者独立存在，却为创立一个单一编码表而协同工作。所以从大体上说，Unicode = ISO 10646。
世界上的所有字符数，大概有几万个。ISO 10646标准所定义的字符集称作为通用字符集Universal Character Set(UCS)。UCS-2表示用2个字节表示一个字符，UCS-4表示用4个字节表示一个字符

Unicode只是定义了每个字符他的编码是多少，但是却没有定义字符应该怎样存储在计算机中。如果Unicode统一规定，每个字符要用三或四个字节表示，那么对于存储来说，是极大的浪费，这样呢，也就形成了Unicode码的多种存储方式。
这种存储方式在互联网上有UTF-8、UTF-16、UTF-32，其中UTF-8是互联网上使用最广泛的Unicode的实现方式，这里，UTF-8只是Unicode码存储的一种方式。
Unicode码的大小和存储的数据的大小(UTF-8)的对应关系

比如说，某个字符的Unicode码表示为15678(1 0101 0110 0111 1000)，那么转换为UTF-8的存储数据之后，就是(11110000 10010101 10011001 10111000),就是按照数据的大小根据UTF-8的编码方式将二进制逐位补充进去，Unicode码最多就是能表示21bit的字符

java中的输入输出流学习

最近写一个处理字符串的程序，读取字符串把我卡了半天，刚学Java的时候，输入输出流就很模糊，现在来补下基础吧
基本概念：
1、流是指一组有顺序的、有起点和终点的字节集合，是一种抽象概念
2、流对程序而言是有方向的，由外部到内部的数据流称为输入流，由程序到外部的流称为输出流
3、一个流要么是输入流，要么是输出流，只能有一种身份。对输入流，只能读，对输出流只能写
4、对流操作的最小单位是字节，也就是每次只能读或写一个字节，为了提高效率，java将一块内存空间设计为缓冲区，将传送的数据都放这里，然后可以一次性的读/写若干字节的数据。缓冲区中的流叫缓冲流
5、当程序读输入流的时候，程序实际从缓冲区中读取，当缓冲区为空时，系统会从相关设备读取数据充满缓冲区。
6、当程序写输出流的时候，先写入缓冲区，当缓冲区满时，系统将数据全部发送到相应的设备，如果说，还没满就要发送，称为立即传输(flush)
7、流按照流中元素的不同分为两类，一类是字节流（每次操作一字节），一类是字符流（每次操作一个字符，2字节的unicode码）

设计模式►建造者模式

设计模式之建造者模式

建造者模式：当要对一个对象进行较为复杂而又固定的处理过程，而处理的过程又可能会有些微的不同来得到不太相同的对象，那么我们如果在主类中对这个对象进行处理的时候，很有可能会落下一步，而且创建对象的过程这些代码会造成大量重复。为了解决这些问题，用建造者模式来解决。

看下组成：
1、对一个对象的处理过程的抽象类
2、根据处理过程的不同创建具体的子类，在子类中实现抽象方法
3、指挥家类，这个类来完成对2中的对象的固定的处理过程，保证了处理过程的完整，也减少了重复代码的书写

设计模式►外观模式

设计模式之外观模式

外观模式，大致可以理解为，一个车间，外界需要某些东西的时候，按某个按钮，车间就去调配各种机器，然后车间返回给我们需要的产品。这里面呢，我们只需要知道车间能做什么就可以了，而不需要去知道里面的机器有什么，我要生产一个东西，需要哪些机器来完成，这样子，将类与类之间的耦合性降低，也使得高层不用直接依赖那些低层的类，这样子设计符合迪米特原则(尽可能的让类与类之间少知道)和依赖倒转原则(高层不直接依赖低层，二者均依赖接口)。外观，想象地说明了，我们只用了解外观是干嘛的，里面怎么实现的我们不用关心。

设计模式之模板模式

模板方法模式：将一个操作中的算法的骨架定义出来，然后将可变部分定义为抽象方法，让继承他的子类去实现，就比如说InputStream类的设计，定义了抽象方法read，还有read(byte[] b)等方法，InputStream类是一个模板，然后继承他的子类就去实现这里的read方法，从哪里读，怎样读，这些就由子类自己发挥，然后在read(byte[] b)这样的方法中去调用这个read()方法，那么不同的子类，读的方法、从哪读都不同。

设计模式►原型模式

设计模式之原型模式

1、原型模式，应用于，当我们需要创建一个类的许多实例，并且初始化的信息相同的时候，如果在调用构造函数时初始化较复杂，那么就会大大影响程序的性能。原型模式是说在类中定义一个可以复制自己的方法，当创建一个对象之后，调用该方法复制自己得到多个同类型对象
2、java的Object类定义了clone()方法，克隆自己得到另一个对象，这里的clone方法是native原生方法，效率远高于java中的非native方法。

原型模式组成：

1、一个原型接口或者抽象类，定义clone方法
2、多个具体原型类，实现1中的clone方法，还有自己的其他一些方法
